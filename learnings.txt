# orb-browser learnings

## The /task vision agent — how it works with Orb (2026-04-02)

### Architecture (correct approach)
- `/task` is SYNCHRONOUS — the HTTP request stays open
- Agent calls `https://openrouter.ai/api/v1/chat/completions` directly
- Orb intercepts outbound calls to `openrouter.ai` (it's in LLM_DOMAINS)
- Orb checkpoints the VM during the LLM call, restores when response arrives
- The client just needs a long timeout (5-10 min)

### What did NOT work
- Background task + polling: Orb checkpoints the VM and doesn't auto-wake for poll requests
- Using OPENAI_BASE_URL proxy: only handles Anthropic /v1/messages format, not /v1/chat/completions
- Short timeouts: Orb proxy avg latency is ~244s for vision calls

### What works
- Synchronous /task with Gemini Flash via OpenRouter: ~5s for simple pages
- Orb transparently checkpoints/restores during LLM calls
- retry logic on httpx connection errors (checkpoint kills TCP)
- CERN article with images: correctly described photos in 4.8s

### Remaining issues
- Some pages cause 500 errors after multiple sequential tasks (GitHub, CERN on 2nd run)
- Likely connection state issues after checkpoint/restore on heavy pages
- Need better error reporting to diagnose

### Model compatibility (OpenRouter)
- `anthropic/claude-3.5-haiku` does NOT support image input (Bedrock provider)
- `google/gemini-2.0-flash-001` works for vision, good quality
- `qwen/qwen-vl-plus` works for vision, cheapest
- Model name format: dot notation e.g. `anthropic/claude-3.5-haiku`

### Orb proxy internals
- openrouter.ai is in LLM_DOMAINS — intercepted at network level
- OPENAI_BASE_URL env var is set but only for /v1/messages (Anthropic format)
- admin.orbcloud.dev shows VM status, checkpoint stats, LLM proxy metrics
- Checkpoint stats: 100% success rate, avg 757ms, avg size 45.2 MB
- Restore stats: 100% success rate, avg wake 507ms, p95 768ms
